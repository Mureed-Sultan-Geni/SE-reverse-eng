Architecture

Here’s a description of the main parts

HTML/PUG

SetJS keeps HTML separate from code. All components are made using HTML chunks. In the boilerplate, you can use either PUG or HTML. The end result is the same. Just make sure the file names without the .html or .pug extension are unique or files will be overwritten. We do not recommend mixing pug and html. Use only one or the other. By default PUG support is commented out in static/gulpfile.js/tasks/views.js. Make sure you enable it there before using it (you’ll also have to run npm i -D gulp-pug before you can use it).

scripts/main.js

Loads the SPA. The boilerplate comes with everything fully configured to call the start function exported by this module. This in turn prepares and loads all the subsystems and tells the SetJS library to initiate routing and page loading. Generally, you’ll not edit this file.

scripts/app

This is where you’d spent most of your time. It contains all the route setup as well as the actual pages of the application. Typically, you’d have separate folders in this directory for modularity e.g. app/account to deal with auth, app/site for general pages, app/dashboard for dashboard and so on. Each folder will have its own pages, routes, API call setup etc.

scripts/bootstrap

This folder contains the initial bootstrapping of the plugins and components the application will be using. Plugins typically do not require any special calls to bootstrap. You will import the plugins from either setjs (SetJS library) or your own plugin folders. This folder has two files:

  • component-init.js Here you initialize your components like header, footer etc.
  • plugin-init.js Here you import plugins from scripts/setjs/plugins (SetJS common plugins) as well as from scripts/plugins (application-specific plugins)

As always, you can add more files to the bootstrapping folder/process.

scripts/components

A place for UI components e.g. footer, nav, sidebars etc. The corresponding HTML should be placed under a similar hierarchy. For example, scripts/components/footer.js will have its corresponding HTML file views/components/footer.html. The boilerplate comes with the correct folder structure for the basic building blocks. You just need to follow the same pattern to keep well-organized.

scripts/config

Provides configuration for other parts of the applications. The pre-existing files enable a well configured system. You can add more files or edit existing ones to suit your needs. The following configuration files are provided by the boilerplate and must always be present in the default setup:

  • app-config.js Exports app-wide configuration. Exports the loader (which provides UI for the page transition process) and a viewUpdate function used by the setjs/kernel/page-loader.js file during the page loading process.
  • event-types.js Event types used by the setjs/kernel/event-manager.js event system. SetJS, according to its philosophy of keeping everything simple, comes with a very concise event system. The purpose of events is to decouple unrelated parts. Use events wisely and sparingly only when they make total sense, so as to make code easier to understand and debug. SetJS does not believe in total decoupling and dependency injection like some other SPA frameworks do.
  • setup.js Used to make sure non-existing configuration is not used by accident. Also provides the templateUrls function (so that you can load your templates from wherever you want). Application settings are provided to main.js on start-up and are converted into functions by this object. It ensures safety against having a badly configured start-app routine.
  • storage-types.js If you want centralized control over certain fields that are stored in localStorage of the browser, you can set them up here for use by the setjs/kernel/storage.js file.
  • user-roles.js SetJS comes with a system of user roles for easy management of page access levels. The actual delivery of content and role based restrictions are the responsibility of the API/Backend (as this cannot be done client-side). Make sure you edit this file to match your BE roles.

scripts/core

Application modules that provide core functionality for other parts of your SPA. Provides a few modules by default. As always, you can add more to suit your SPA needs. These come bundled with the boilerplate:

  • api-helper.js Helper class for a consolidated system of communication with the backend API(s). Exports an api object, so that other parts can expose their API through this single object. This helps simplify how you use your API calls. For examples of good API setups, please check the available demo projects (TODO: links to demo pages).
  • app-data.js Initializes application data at start-up. Also creates a defData object, which is used by SetJS to provide global data for the views. By convention, all global properties exposed for views use the `@` prefix, such as `@route` (the page route object), `@lang` (the current language file) and so on.
  • asset-loader.js Responsible for loading Template, CSS and JS files requested by the page object through its asset property or getAssets function. You can modify this file to suit your needs (you’d rarely, if ever, need to do that though).
  • events.js Basic application events including throttled scroll and resize events. You can comment out the default scroll and resize hooks if not needed.
  • keyboard.js Provides some common keyboard hooks. Not an essential module. The defaut boilerplate does not use it. Safe to delete if you do not need it.
  • route-manager.js Manages route handling. Also sets up the browser’s beforeunload event, if required. Edit this file to suit your needs.
  • setjs-init.js Configures the SetJS environment before it can be started by main.js. Provides all the necessary configuration of setjs/kernel/setjs.js (SetJS library).

scripts/entities

This is for your applications data entities and their management. All entities are to be initialized by the core/app-data.js module detailed above. Does not have anything by default. Can be deleted if your project does not need it.

scripts/helpers

Your application’s common helpers. Create as many files and folders as you deem fit.

scripts/plugins

Your application plugins. Has four sub-folders by default to match setjs/plugins plugin structure. Feel free to edit any way you like.

scripts/setjs

This is the SetJS library. Provides the backbone for your SPA and has a well configured and thought-out pragmatic setup. Usually, you would not want to edit anything in there, so you can easily update to the latest versions of SetJS as they are released. Generally, if you wanted different behavior from some module in that folder, you’d create a replacement module outside of it in the application space. However, the rule is not written in stone and is totally based on your application needs. Like every other part of SetJS SPA system, pragmatism prevails. The details of setjs will be provided elsewhere. setjs/utility has some very useful utility modules often needed by your application code. It ensures that, in general, you do not need third party utility libraries like lodash etc.
